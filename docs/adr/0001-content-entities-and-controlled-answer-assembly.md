# Content entities and controlled answer assembly

Use published Sanity **Content Entities**, entity fields, Portable Text blocks, **Claims**, hard constraint rules, and controlled **Answer Components** as the source material for GuideSite answers. The LLM is not the source of truth or the content model; it acts as a controller assistant that retrieves approved source material, selects valid components, and creates an **Answer Composition** under **Composition Rules**.

This rejects the earlier "content atoms" model. Sanity already provides structured, addressable content, so atomicity should be a usage constraint where needed rather than the primary ontology. Entity identity is its domain type, such as Program, Session, Policy, Protocol, Testimonial, Concern, or Claim; trust, proof, role, presentation form, and semantic topic may be metadata or relationships, but they do not define the entity ontology.

For MVP, **Concern** is a lightweight Sanity entity used as the retrieval bridge from a **Prompt** to related **Content Entities**. **Claims** are first-class only when assertions are sensitive, strategic, trust-relevant, or reusable; simple facts should be derived from entity fields. Any source used in an Answer Composition requires a citation, not only Claims. Entity types map to approved Answer Component families, components declare required fields, and rules constrain which components can be used.

Consequences: deterministic logic such as age eligibility, availability, pricing, capacity, deadlines, fit checks, and sensitive escalation belongs in code, but it does not generate complete answers. The AI planner composes answers from approved source material, while composition rules constrain the planner where hard requirements exist. Editorial and semantic relationships such as related concerns, claim sources, testimonial associations, policy references, and guide associations belong in Sanity; orchestration constraints such as approved component use, citation requirements, and sensitive-answer limits belong in the controller layer.

The candidate Ultimate Camp Website content entity catalog includes Program, Session, Policy, Protocol, Procedure, Staff Standard, Training Standard, Testimonial, Camper Story, Parent Story, Important Date, Transportation Route, Requirement, Camper Rule, Parent Requirement, Accreditation, Guide, Checklist, Printed Material, Output, Brand Promise, Concern, Claim, and CTA. This list preserves the architecture exploration but is not exhaustive; the Sanity schemas remain the authority for the implemented catalog.

Printed materials and media entities need an AI-readable **Content Map** in their schema when relevance cannot be reliably inferred from structured fields or text. The system cannot decide whether a brochure, PDF, image, video, audio clip, or other opaque asset is relevant to an answer unless the source entity exposes a distilled description of what it contains and how it may be used.

Content Maps describe contents and relevance; they do not need a separate restriction model by default. The AI planner should use judgment when composing answers from approved source material. If concrete restrictions emerge, they should be expressed as **Composition Rules** or fields interpreted by those rules.
